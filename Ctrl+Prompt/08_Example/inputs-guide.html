The ingredients are everything

Before writing a single prompt, you need to prepare the context. This folder is what gives v0 enough information to generate something convincing on the very first attempt.

📁 6 folders 🧠 Business context 🎨 Design system 📊 Real dataset 🌤 Live API
1
Why invest time in inputs?

Most people open v0 and type: "build me a hotel dashboard". The result is generic — no real data, no brand colors, no business logic.

The difference between a prototype that closes a deal and one that doesn't lies in the quality of the inputs. When v0 receives real business context, a design system with exact colors, and real client data — it generates something the client can look at and say: "this is us".

This process isn't exclusive to vibecoders. Anyone in product, business, or design can build these inputs. Modern AI tools make it accessible to everyone.

2
How we built these inputs — the real process

This is the flow we followed to prepare the STAYSelect inputs. In a real client project, each step is supported by conversations, recordings, and AI tools.

1

Discovery call with the client

A 30–60 minute call where you listen for the business problem, the user, the current pain point, and company context. You don't need to be technical to run this call — you need to ask the right questions.

📞 Zoom / Meet 🎙 Grain / Otter.ai (recording) 📝 Your own notes
2

Transcription + generative AI for the Context doc

With the call recording, you use Claude or ChatGPT to generate the first draft of the Context doc. Feed it the transcript and ask: "extract the company, the problem, the user, and the prototype scope". Review and adjust in 10 minutes.

🤖 Claude / ChatGPT 📄 Call transcript ✏️ Human review
3

BRD with AI — the business "why"

With the Context doc ready, you ask Claude to generate the BRD. The key is giving it context + the client's business objectives. The BRD speaks to the Director or VP who makes the purchase decision — business language, not technical.

🤖 Claude 📋 Context doc 💼 Review with sales team
4

PRD — specifying what to build

The PRD defines exactly the screens, columns, calculations, and flows. You can dictate it, record a video walkthrough, or write it — then AI structures it. The key is being specific: what columns the table has, how KPIs are calculated, how the user navigates.

🤖 Claude 📹 Screen recording of desired flow 🗣 Voice dictation
5

Design System — palette, components, and style

Take the client's logo, their brand colors, and describe the desired visual style. AI can generate the full design system with Tailwind tokens and shadcn/ui component mapping. You don't need to be a designer.

🎨 Client logo 🤖 Claude 🖍 Visual references (screenshots, URLs)
6

Dataset and APIs — real data

Get real data from the client or generate a representative dataset. For public APIs like Open-Meteo, simply document the endpoint and response mapping. This is what makes the prototype look alive — not generic.

📊 Client Excel / CSV 🌐 Free public APIs 🤖 AI for generating mock data
7

Features — prioritizing the prototype MVP

List everything the prototype could have and classify it: P0 (non-negotiable for the demo), P1 (adds WOW), P2 (nice to have). This keeps you focused on what matters for closing the deal.

📝 Prioritized list 🤖 Claude for structuring 🎯 Criterion: does this close the deal?
Real time to build these inputs: For STAYSelect, the entire input process took approximately 2 hours — including case definition, context documents, BRD, PRD, design system, and features. With generative AI and a good dataset, this time is dramatically reduced compared to doing it manually.
3
The 6 folders and their artifacts
🏨
01_Context — Business context
00_Inputs/01_Context/
The most important artifact. Without clear business context, v0 generates something generic. With a good context doc, the first output already looks like the client's product.
  • Who STAYSelect is, where it operates (Sunset Hotel, Cancún)
  • The core problem: vendor lock-in, no visibility into their own data
  • The user who will use the dashboard (Operations Manager)
  • The key sales message: "your data, your tool"
  • Exact prototype scope: what's IN and what's OUT
📄 stayselect-context.md
📋
02_BRD_PRD_Specs — Requirements
00_Inputs/02_BRD_PRD_Specs/
Two separate documents with distinct purposes: the BRD speaks to the business (the "why"), the PRD speaks to the product (the "what"). Together they give v0 all the specification it needs to build something correct.
  • BRD: business objectives, stakeholders, scope, risks, and acceptance criteria
  • PRD: specification of the 3 screens with exact columns, KPI calculations, and navigation flow
  • Technical constraints: read-only, no login, local CSV, 1440px
📄 BRD.md — Business Requirements Document 📄 PRD.md — Product Requirements Document
🎨
03_UXUI — Design system
00_Inputs/03_UXUI/
Defines exactly which shadcn component goes where, with what colors and in what layout. Without this, v0 uses its own defaults and the prototype doesn't look like the client.
  • Full palette: Primary #1A6FA8, Accent #E8622A with Tailwind tokens
  • shadcn/ui component map per screen (Cards, Table, Select, Badge, etc.)
  • Dashboard grid layout: exact rows and columns
  • Weather widget: inline visual specification in the header
  • Design principles: enterprise but human
📄 design-system.md — Complete design system
📊
04_Data_Sources — Real dataset
00_Inputs/04_Data_Sources/
A prototype with real data is infinitely more convincing than Lorem Ipsum or fictional data. This CSV has over 36,000 real hotel reservation records — numbers, dates, and patterns that tell a real story.
  • 36,000+ reservation records from Sunset Hotel
  • 19 fields including dates, prices, statuses, channels, and room types
  • Real mix of canceled and non-canceled (~37% cancellation rate)
  • Multiple channels: Online, Offline, Corporate, Aviation, Complementary
📊 Hotel Reservations.csv — Sunset Hotel dataset
🌤
05_APIs — Real-time integration
00_Inputs/05_APIs/
The weather widget is the detail that makes the audience say "is that real?". A free, no-registration API that shows real Cancún data in the header — transforms the prototype perception from "mock" to "live application".
  • Open-Meteo API: free, no API key, CORS enabled
  • Exact Cancún coordinates: 21.1619° N, -86.8515° W
  • WMO weather code to lucide-react icon mapping
  • Silent fallback: if it fails, the app doesn't break
  • Demo talking point included for when the audience notices
📄 README.md — API documentation and implementation snippet
✅
06_Features — Prioritized list
00_Inputs/06_Features/
Without prioritization you get lost in features that don't matter for the demo. P0 is what can't be missing to close the deal. P1 adds WOW. P2 is for if there's extra time. This list keeps focus during the build.
  • P0 (non-negotiable): KPI cards, bookings chart, table, detail view, branding, weather widget
  • P1 (WOW factor): channel donut chart, room type bars, date filter
  • P2 (nice to have): lead time histogram, frequent guest badge
  • Each feature mapped to the exact CSV field that feeds it
📄 features-prototipo.md — Features P0 / P1 / P2
